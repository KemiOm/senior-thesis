================================================================================
1. HOW RESULTS WERE GATHERED (END-TO-END)
================================================================================

1a. Data anchor
  - Fixed poem splits: evaluation/splits/{train,dev,test}.json (seeded partition).
  - Task rows: output/training_data/<task>/test.json (and train/dev for SFT).
  - Each eval JSON stores: model id or local path, prompt_type, prompt_style, task,
    n_samples cap, and a list of per-line records (input, gold_target, prompt,
    model_output).

1b. Inference (generation)
  - Command-line driver: scripts/run_prompt_eval.py
  - Seq2seq (FLAN-T5): encoder receives the prompt; decoder generates the answer string.
  - For merged LoRA checkpoints saved as directories, sft_eval.slurm AUTO-selects
    --prompt-style line_only when --model points at a local directory (unless
    PROMPT_STYLE is set explicitly). That means the prompt is EXACTLY the training
    JSON "input" field (raw line, continuation context, or [start] / pipe-joined
    context for combined) — NO long instructional header and NO few-shot quatrain.
  - Hub baseline rows in results/model_comparison.csv use slug hub/google_flan-t5-large
    with prompt_style=default and prompt_type=zero_shot (full instruction templates
    from PROMPTS in run_prompt_eval.py).

1c. Where files live
  - SFT / round eval JSON: results/round*/.../*.json (relative paths in CSV).
  - Pretrained copy for side-by-side: results/hub/google_flan-t5-large/*.json
  - Round 3 also duplicates hub eval under results/round3/google_flan-t5-large/.

1d. Rollup to a single table
  - evaluation/scoring/rollup.py collect_rows() reads every *.json under the chosen
    root (here: results/), skips paths containing "_archive".
  - For each file it computes:
      * exact_match_pct — character-identical pred vs gold after a small normalize
        (rollup.normalize_for_task; whitespace collapse for natural_text only).
      * Levenshtein statistics (edit_distance.aggregate_string_edit_metrics):
          - Gold and pred are passed through normalize_for_eval(task, ·):
              meter_only: stress_normalize_for_compare (0/1 → +/-)
              rhyme_only: whitespace-normalized, casefold
              combined: strict parse parse_combined_bundle, else loose
                parse_combined_bundle_loose, then canonical_combined_for_compare
          - Per line: normalized edit distance / max(len(gold),len(pred),1)
          - levenshtein_similarity_pct = 100 * (1 - mean(normalized distance))
      * Structured extras (include_structured_partial=True in collect_rows):
          rhyme_only → st_rhyme_relaxed_match_pct (case/space insensitive token match)
          combined → st_combined_* (parse rate, per-field match %, mean field agreement)

  - Output written to results/model_comparison.csv (same schema as evaluation
    baseline_report, but different directory to avoid overwriting pretrained tables).

================================================================================
2. EVALUATION TASKS (WHAT EACH COLUMN / JSON MEASURES)
================================================================================

meter_only
  - Input: one line of verse (normalized text).
  - Gold: +/- stress string (minimum length rules in data prep / strict eval).
  - Primary readout: exact_match_pct on exact +/- strings; similarity after stress canon.

rhyme_only
  - Input: same line text.
  - Gold: ARPAbet tail from last stressed vowel, rhyme group fallback, or "none".
  - Rollup adds st_rhyme_relaxed_match_pct (usually tracks near exact for clean outputs).

combined
  - Input: continuation-style context for the NEXT line ([start], one line, or " | ").
  - Gold: stress:X|meter_type:Y|rhyme:Z|end:E|caesura:C
  - Full-string exact match is extremely strict; use st_combined_mean_field_pct and
    st_combined_pred_parse_pct for partial structure. Combined rows in the CSV show
    near-100% pred_parse for models that emit parseable bundles even when wrong.

================================================================================
3. PROMPTS FOR FINE-TUNED (MERGED) MODELS IN THIS ROLLUP
================================================================================

For every row with json_path ending in *_line_only.json:
  - prompt_style = line_only
  - prompt_type = zero_shot (filename convention; IMPORTANT: there is NO separate
    few-shot example block — the model sees ONLY the raw input string, matching the
    SFT training format in output/training_data/<task>/*.json.)

For hub/google_flan-t5-large rows:
  - prompt_style = default
  - prompt_type = zero_shot
  - Full natural-language instructions + explicit output constraints (see PROMPTS
    dict in scripts/run_prompt_eval.py).

Default Slurm grid (scripts/hpc/sft_eval.slurm):
  - PROMPT defaults to zero_shot
  - N defaults to 500 test lines per task
  - SPLIT defaults to test
  - 4 models × 3 tasks = 12 runs: BASE (Hub id) + three merged LoRA directories

  - Report Round 3 learning-rate 1e-4 adapters as the PRIMARY fine-tuned checkpoints

  - Use Round 2 or Round 3 combined as the joint model depending on whether you want
    the slightly newer combined timestamp (R3 combined vs R2 combined are very close

================================================================================
8. INFERENCES AND CONCLUSIONS (DEFENSIBLE FROM THIS CSV + SCORING CODE)
================================================================================

8a. Supervision specialization dominates
  - Large gains on exact match for meter and rhyme occur ONLY when the eval task
    matches the training JSON objective (diagonal of the heatmaps).
  - The same merged weights score near-zero exact on the other single-task JSON
    formats, with similarity often collapsing (meter→rhyme) or behaving like an
    unrelated string mapping (rhyme→meter).

8b. Pretrained instructional prompting ≠ SFT input-only evaluation
  - Hub rows (default prompts) are not directly comparable numerically to line_only
    merged checkpoints; thesis text should separate “cold-start Hub behavior” from
    “merged model on training-distribution prompts.”

8c. Combined objective
  - Full-bundle exact match stays at 0% in this table, but Levenshtein similarity
    is high (~84%) and mean per-field overlap is mid-30% with universal pred_parse —
    models learn a bundle FORMAT and partially align fields without nailing every
    token of the gold string.

8d. Learning rate (Round 3 vs Round 4)
  - With all else held equal in naming, lr=1e-4 beats lr=5e-5 on specialist exact
    match for these runs — report as a small hyperparameter study, not a universal law.

8e. Scientific takeaway aligned with OVERVIEW.MD
  - The experiment successfully decouples “which formal signal is trained” from
    “which evaluation probes run,” showing that FLAN-T5-Large can absorb verifiable
    formal targets via LoRA, that skills are largely non-transferable across these
    three different output alphabets without additional training, and that joint
    training produces a different error profile than single-task experts (use
    compositionality / transfer figures to quantify).

10. FINE-TUNING HYPERPARAMETERS — WHAT CHANGED, WHY, AND HOW
================================================================================
Canonical copy also lives at: results/results.txt

Ground truth: scripts/sft/lora_train.py, scripts/hpc/lora_train.slurm, and
run_params.json under sft_runs/ (cluster jobs pass LR, EPOCHS, etc. via env).

--- What stayed mostly fixed (controlled comparison) ---
Same backbone: google/flan-t5-large.
Same task JSON: output/training_data/<task>/{train,dev}.json.
Optimizer shape: weight_decay=0.01, warmup_ratio=0.1 (Trainer via script).
LoRA dropout: 0.05. Seed: 42.
Per-device batch: train 8, eval 4 (unless reduced for OOM).
merge_and_save: true → final_model_merged/ for eval with run_prompt_eval.py.

Goal: the scientific lever is supervision target (meter_only / rhyme_only / combined),
not a new architecture every run.

--- Round 1 → Round 2: changes and rationale ---
Round 1 (exemplar: sft_runs/round1/combined/run_params.json, 2026-04-11):
  num_train_epochs 3; learning_rate 1e-4; lora_r/alpha 16/32; targets q,v;
  max_input_len/max_target_len 256/64; gradient_accumulation_steps 1;
  eval/save 1000; no_fp16 true, bf16 false (FP32 on GPU).

Round 2 (sft_runs/round2/{meter_only,rhyme_only,combined}/run_params.json, 2026-04-13):
  epochs 3→5; LR 1e-4→5e-5; LoRA 16/32→32/64; targets q,v→q,k,v,o;
  caps 256/64→384/128; grad_accum 1→2; eval/save 1000→300; still FP32 train.

Rationale: more capacity and length for structured/combined outputs; lower LR +
more epochs to stabilize; finer checkpoints. How: Slurm/CLI overrides (LR, EPOCHS,
LORA_R, GRAD_ACC, --max_input_len, --max_target_len, --lora_target_modules).

--- Round 3–4: LR sweep ---
Round 3 (e.g. sft_runs/round3/meter_only_lr1e4 vs lr5e5/run_params.json, 2026-04-15):
Restored smaller LoRA (16/32, q,v), shorter caps (256/64), grad_accum 1, eval/save
1000; kept 5 epochs; swept learning_rate 1e-4 vs 5e-5; switched to bf16 training
(no_fp16 false, bf16 true).

Why: Round 2 bundled many changes—hard to attribute. R3–4 isolates LR on a
near-default adapter recipe. Downstream: results/model_comparison.csv favors
lr=1e-4 over 5e-5 on specialist exact match for these checkpoints.

--- One-paragraph thesis snippet ---
We kept FLAN-T5-Large and the JSON supervision fixed, used PEFT LoRA with fixed
seed and standard weight decay / warmup, and first explored a stronger schedule
(more epochs, lower LR, higher LoRA rank, additional target modules, and longer
token caps) in Round 2 to better fit long structured targets; we then restored a
default-like adapter configuration and varied only learning rate (1e-4 vs 5e-5)
in Rounds 3–4 to obtain an interpretable hyperparameter conclusion supported by
the held-out eval rollup (results/model_comparison.csv).
