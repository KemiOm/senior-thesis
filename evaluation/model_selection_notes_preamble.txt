# What these baselines measure

Each row in `evaluation/results/model_comparison.csv` is one **prompt-only** run: a model was given the same test lines as in `output/training_data/<task>/test.json`, and its string output was compared to the **gold target** for that task.

## Tasks (four conditions)

- **meter_only** — Gold is a stress pattern (e.g. `+-+…`). The model must reproduce that pattern exactly for a line to count as correct.
- **rhyme_only** — Gold is a rhyme label (phonetic key from the pipeline). Again, the whole string must match exactly.
- **natural_text** — Input is prior line(s) in the same poem (or `[start]`). Gold is the **next line** (plain text). Exact match is strict after whitespace normalization.
- **combined** — **Same input as natural_text.** Gold is the **next line’s** bundled string (`meter:…|rhyme:…|end:…|caesura:…`). JSON rows may include `next_line` (surface text of that line) for tooling; scoring still uses `target`.

So across tasks you are always asking: **did the model output the same label (or line) the corpus says is correct?** This is intentionally harsh: one wrong token in a pattern usually counts as a full miss.

## Primary score: exact match %

**Exact match %** = (number of lines where `model_output` equals `gold_target` after task-specific normalization) ÷ (number of lines with non-empty gold) × 100.

That number is what the “Best by task” and “Best mean” sections below use. It is the headline metric for comparing models on the same task and same sample size (`n_scored` in the CSV).

## Extra columns in the CSV (optional)

If ran the summarizer with extra flags:

- **`st_*` (structured partial credit)** — For **rhyme_only** and **combined**, these relax strict equality a bit: they check whether normalized rhyme tokens or individual parsed fields match, even when the full string does not. Use them to see whether the model is “close” on one part of the target.
- **`nt_form_*` (natural_text form)** — These run CMU-style phonology on both gold and model text. They only apply to rows where **both** sides are parseable; when almost every generation fails that path, the percentages are not stable for ranking. Prefer **exact match** and the structured tasks for model choice unless `nt_form_evaluable` is large.

## How to read the auto-generated sections below

- **Zero-shot** often scores **0% exact** on meter/rhyme/combined because the output format is hard to guess without an example. That does **not** mean every model is equally bad for fine-tuning; it means exact match is the wrong lens for that prompt style.
- **One-shot** and **few-shot** usually matter more: the model sees the label format (or a short example), so exact match becomes more informative.
- **Best mean** averages exact match across the four tasks for models that have **all** four JSONs. It favors generalists; if you only care about meter, sort on **meter_only** instead.
- **Incomplete folders** lists models missing a task file (often a job that did not finish or was not submitted for that task).

When the script suggests a default SFT candidate, it uses **few-shot first, then one-shot, then zero-shot** so the pick is not driven by all-zero zero-shot ties.
